# 简历

其实简历的模板以及如何写好简历在网上有一大堆相关文章，所以我就单挑几个比较重点的方面说下，另外不同阶段的简历肯定侧重突出的是不一样，我这里只说下校招以及3年工作经验的情况
（为什么没有更往后的？因为我也不是什么老资历🤔 就不瞎扯以防误导别人   

## 校招或者1年以内的简历

这个阶段其实是一个成长的时间，大多数人可能是一两段的实习经历或者短短的一阵正式工作，这阶段是很难考察出在业务上有什么个沉淀以及一些业务方案，
所以这个时候往往更会看重基础以及对技术的热情，有没有对最近的热门技术方案有所了解甚至实践

+ 基础知识
+ 如果没有实习或者工作，是否有自己亲手写的一些小项目，或者是参与社区开源的项目
+ 如果有实习工作，实习以及工作的内容是否有或者能反映出自己独立的思考以及动手能力
+ 记录自己的技术博客，用输出倒闭自己的输入
+ 了解社区的前沿技术，技术趋势

## 1年-3年

这个时候大多都能够独立负责一个或多个项目，需要继续往下钻研探索，避免一年经验重复干两三年

+ 平时常用的框架/库，进阶一步去使用，比如它的一些高级用法是否有所掌握，有没有试着去了解它的原理实现
+ 日常的业务开发中不局限于完成功能，是否有去思考项目结构如何设计
+ 在日常的业务开发中有没有去思考团队提效的方式（比如最近AI的使用层出不穷，是否写过MCP或者相关Skill进行提效，有没有使用SDD相关的工具做任务流规范AI开发的过程
+ 工程化相关的了解以及在项目中的运用
+ 软技能以及对业务的理解。沟通协作也是很重要的一项能力，通过思考业务真实需求砍掉多余的需求，协调各个角色一起推进目标，也是工程师很重要的技能。

## 简历结构

我个人喜欢写以下几个方面，校招实习的时候控制简历在1页，社招的时候控制自己的简历在2-3页，保证相对精简并且能够突出自己就好（比如我学历不好，在校成绩也一般，更没参加过啥竞赛比赛，
我就只会在个人信息里简单写自己的学历以及学校，然后学校的其他方面一概不提了）

+ 个人信息（名字、年龄、多少届、学历、是否在职、联系方式、个人站点等等等等）
+ 工作经历 / 教育背景
+ 项目介绍（STAR法则，项目的背景，你负责了什么，又做了什么，最后得到一个了一个怎样的结果）
+ 优势总结（或者说是专业技能，可以列一下自己有什么样的能力或者经验，可以使用xxx、xxx 做 xxx 之类的，这一块最好跟你投递的岗位JD匹配） 

另外，如果简历上提到的专业名词或者技能，至少自己一定得要对其有所了解，不止于简单的使用，不要抱着面试官可能不会问到的侥幸心理

## 突出结果

在项目经历里除了表现自己的职责以及方案以外，得到了一个怎样的结果也是尤为重要的，而这个结果最好是有数据作为量化，而不是一个单纯的体感 

最简单的例子，如果你是没有工作经历找实习，自己写了个小项目，项目里可以体现你自己封装了x个组件，复用于x个页面等，又封装了x工具用于提升怎样的效率，做了什么实现了怎样的功能

再比如有实际项目的性能优化，使用了xxx方案去优化了一个页面的加载，那么可以体现在FCP、LCP等数据提升了多少，然后也要清楚为什么以这些指标的变化去作为衡量这件事情的结果，使用某个指标的原因往往跟你要解决的问题挂钩  

除了技术上的指标结果，当然也可以写业务上的指标结果，例如多少的PV、多少的转换率等等，但是自己就得准备好相关的追问，例如如果写了庞大的PV，那么面试官说不定就会对你在性能的优化或者容灾方面感兴趣

# 面试

虽然整个招聘的流程是把自己推销出去的一个流程，但是在面试阶段，我更鼓励把这件事放在一边，当作一个一对一的沟通，面试的过程就不会是单纯的你问我答，
所以积极地去表达自己，无论是回答、思路还是自己的疑问

## 讲好故事

要知道在面试的时候双方的信息是非常非常的不对等的，所以你项目里的背景、遇到的问题、得出解决方案的过程、你的思考顾虑以及最终的实施，整个过程以及细节里面面试官是一片空白的，
所以把一个故事讲好除了让面试官了解你，也是体现出自己具有较好的表达能力的体现，不要想着自己简单讲两句，接着就等面试官提问就好，有时候讲的太笼统了面试官也不知从何问起，
尽量让你和面试官之间的沟通有一个好的开头

## 引导面试

👆上面已经说了要讲好一个故事，那么一个好的故事你就得学会埋伏笔，在介绍自己项目的时候适当的留下一些疑问  

举个简单的例子，实现一个功能的时候可以说自己调研了 x 项目的实现以及 y 项目的实现，
但是在介绍的时候不要说自己调研出来了个什么结果以及他们之间的对比，直接说自己最终参考了 x 项目的实现方式，至于为什么选择 x 项目的实现方式这一块可以留空出来等着面试官的提问，
自己就可以顺延下去介绍 x 跟 y 实现的区别以及他们得到的结果，基于你们自己的业务场景，你觉得哪种才是最适合当下的

上面这个只是简单的例子，重点是引导面试官顺着自己的脉络去发问，从而把控一定的主动权以及自己的节奏

## 拓展思路

既然是一场交流，那么肯定多少也是会有碰到自己模糊不清的理念或者技术点，在自己不够清楚面试官的意图或者表达的时候，都可以勇敢地进行提问，
确定面试官真正期望的你们之间的交流方向是什么，如果实在不明白，也可以大胆的去说自己对这个话题的见解以及自己能想到的解决方式；
除非是硬性的知识点，一件事情的解决方案往往不会唯一，可以大胆的去拓展自己的思路，让对方看到你对这个话题的思考，这样总比干瞪眼好

## 反问

这里我觉得挑自己喜欢的话题来提问就好，最好是能够具体一点，不要提问太泛的问题，因为太泛的问题得到的答案肯定也是空泛的，一场面试也是难得的增长见识以及经验的机会，
我们尽量提升它的含金量 

可以把之前工作上遇到的一些问题脱敏后听听对方的解决方案以及思考，也可以问问对方业务需求实现的一整个工作流，
看看对方所在的部门的一个工作流程，或者了解下所面试的部门要负责的事情，在企业近两年的战略中处于什么样的位置，为什么要去做这些事情 

# 最后
